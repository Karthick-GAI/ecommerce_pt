AI E-Commerce Platform — Capstone

13 decisions. Deliberately made.

Every architectural choice had alternatives. This document states what we chose, why we chose it, and exactly what we gave up.

13 Decisions documented

01 Data Layer
D01
VECTOR_STORE = pgvector
Chosen ✓

Product embeddings (1536-dim) live in a Vector column on the PostgreSQL products table via the pgvector extension. The decisive advantage is query composability: hard SQL filters — price cap, in-stock flag, category — execute in the same query as the ANN ranking using the <=> cosine operator. A dedicated vector DB would require a two-phase approach (pre-filter in SQL, re-rank in vector DB), adding a network hop and a consistency gap between the two stores.

⚖ Trade-off accepted pgvector is not optimised for millions of vectors across tenants. A production catalogue exceeding ~500K products should migrate to a managed vector service. The embedding generation code does not change — only the storage backend.
Rejected Pinecone × Weaviate × Qdrant × ChromaDB ×
D02
DATABASE = postgresql + sqlite
Chosen ✓

Twelve business-domain services share a single PostgreSQL instance — required for pgvector and cross-service analytics joins. The two auth services, user_management and seller_portal, run on isolated SQLite databases, keeping credential data physically separated from product and order tables without the overhead of a second PostgreSQL instance for the capstone.

⚖ Trade-off accepted SQLite does not support concurrent writes. Under high registration load the write-lock becomes a bottleneck. Production fix: promote auth stores to PostgreSQL schemas with PgBouncer connection pooling.
Rejected MongoDB × Per-service PostgreSQL × Single shared SQLite ×
02 AI & Models
D03
LLM_MODEL = gpt‑5.4‑mini
Chosen ✓

All LLM tasks — RAG generation, NL query parsing, multi-agent reasoning, guardrail judging — use gpt-5.4-mini on Azure OpenAI. It hits P95 RAG latency of 2.3s (within the 3s NFR target) at roughly one-fifth the cost of GPT-4o, and produces reliable structured JSON without brittle post-processing. Azure deployment keeps data in-region and provides an enterprise SLA.

⚖ Trade-off accepted Azure OpenAI vendor lock-in: every AI call depends on Azure availability. Mitigated by the circuit breaker — if Azure is unavailable, the shopping assistant returns raw vector-search results without GPT augmentation.
Rejected GPT-4o (5× cost, marginal gain) × GPT-3.5-turbo (weaker JSON) × Llama 3 self-hosted (no GPU) ×
D04
EMBEDDING_MODEL = text‑embedding‑3‑small
Chosen ✓

text-embedding-3-small at 1536 dimensions costs $0.02 per million tokens — 6.5× cheaper than ada-002 with slightly better MTEB quality (62.3% vs 61.0%). Embedding all 5,000 products cost under $0.01. The 2.3-point gap between 3-small and 3-large was indistinguishable in user-facing search results for a single-language English/Hindi catalogue.

⚖ Trade-off accepted text-embedding-3-large scores 2.3 MTEB points higher. In cross-lingual or highly ambiguous query scenarios, this gap becomes visible. The cost differential (6.5×) outweighs the quality gain at capstone scale.
Rejected text-embedding-3-large (6.5× cost) × ada-002 (worse + more expensive) × Sentence-transformers local ×
D05
RECOMMENDER = hybrid_adaptive
Chosen ✓

Three signals are blended with weights that shift by user maturity: user-based collaborative filtering (purchase graph), content-based similarity (pgvector cosine on product embeddings), and trending scores. Cold-start users receive trending-heavy results; active users (5+ purchases) receive CF-heavy results. Leave-one-out cross-validation on 50K orders: hybrid adaptive Hit@10 = 0.78 vs CF-only = 0.67 vs fixed-weight hybrid = 0.74.

⚖ Trade-off accepted Weight thresholds ("5+ purchases = active") were set heuristically. The item-similarity matrix is pre-computed in a manual batch job — recommendation quality degrades as the purchase graph grows stale between runs. A production system would run nightly recomputes and learn weights via bandit or A/B testing.
Rejected Pure CF (cold-start fails) × Pure content-based × Fixed-weight hybrid × Neural CF (no training pipeline) ×
03 Service Communication
D06
SERVICE_COMMS = sync_http
Chosen ✓

All inter-service calls use synchronous HTTP via httpx with a 2s timeout. The primary user journeys — checkout, product search, recommendation — are request-response flows where the UI is waiting for a result. Async messaging adds latency and operational complexity without reducing it for these flows. The circuit breaker fast-fails after 5 consecutive downstream failures, containing cascade without a message broker.

⚖ Trade-off accepted Tight temporal coupling: if inventory_service is slow, checkout is slow. In production the order → payment → fulfilment flow belongs on Kafka topics (order.created, payment.confirmed) to decouple services and guarantee delivery under failure.
Rejected Kafka × RabbitMQ × gRPC × Celery + Redis ×
D07
MESSAGE_BROKER = none
Chosen ✓

No broker is deployed. The one flow that genuinely requires async delivery — payment confirmation — is handled by Razorpay and Shiprocket webhooks: they push HTTP POST on state change, which is already an async push model. Adding a broker would insert an extra hop without improving the delivery guarantee. The payment_id is the natural idempotency key for webhook retries.

⚖ Trade-off accepted If Shiprocket is slow, the webhook handler blocks, and Razorpay may retry. In production the handler would publish order.paid to Kafka, return 200 immediately, and a consumer would handle Shiprocket booking with configurable retries.
Rejected Kafka (Zookeeper overhead) × RabbitMQ × Azure Service Bus ×
04 Reliability & Resilience
D08
CACHE_STRATEGY = db_persisted_targeted
Chosen ✓

No distributed cache was introduced. Three targeted patterns replace it: (1) Idempotency cache — completed checkout responses persisted in checkout_idempotency_keys (PostgreSQL), committed atomically with the Order row so correctness survives a service restart. (2) Product embeddings — pre-computed at ingest, reused at every query. (3) Circuit breaker state — in-process memory per service instance. PostgreSQL P95 query time held below 50ms at 100 concurrent users in load testing.

⚖ Trade-off accepted In-process rate-limit counters and circuit breaker state do not aggregate across replicas. At 50K concurrent users, Redis becomes mandatory for shared counters and hot-catalogue page caching. slowapi is already configurable with storage_uri="redis://..." when deployment grows.
Rejected Redis response cache × Memcached × Redis for idempotency (non-atomic with DB) ×
D09
CIRCUIT_BREAKER = in_process
Chosen ✓

Circuit breaker state (CLOSED → OPEN → HALF_OPEN) lives in-process using threading.Lock and module-level singletons in nfr/circuit_breaker.py. Each service registers named breakers per downstream dependency. After failure_threshold (default 5) consecutive failures the circuit opens and fast-fails further calls. After recovery_timeout (default 30s) a single probe request is attempted; success closes the circuit, failure re-opens it.

⚖ Trade-off accepted State does not survive process restart and does not aggregate across replicas. Two instances of checkout_service each maintain independent breaker state — one could be OPEN while the other is CLOSED. Production fix: externalise to Redis using pybreaker with a Redis storage backend.
Rejected Redis-backed pybreaker × Resilience4j (JVM) × No circuit breaker ×
D10
IDEMPOTENCY = db_persisted_header
Chosen ✓

Checkout accepts an Idempotency-Key request header. The key, endpoint, serialised response body, and status code are written to checkout_idempotency_keys in the same PostgreSQL transaction as the Order row. Replaying the same key returns the original response — correct even after a service restart because the record is durable in the DB.

⚖ Trade-off accepted Every new checkout adds a DB write for the idempotency record. This is intentional: atomicity with the order row is non-negotiable for financial operations. A Redis TTL key would be faster but risks losing the record before AOF flush, breaking the guarantee exactly when it matters most.
Rejected Redis TTL key × No idempotency (duplicate orders) ×
05 Platform Design
D11
SERVICE_GRANULARITY = 13_microservices
Chosen ✓

13 independent FastAPI services, each with its own port and requirements.txt, map to distinct business capabilities with separate data ownership and scaling profiles. AI-heavy services (shopping_assistant, multi_agent_system) are CPU/memory-bound; transactional services (checkout_service, inventory_service) are IO-bound. Separate deployment enables separate scaling targets.

⚖ Trade-off accepted Operational complexity: 13 processes to start, monitor, and debug locally. No API gateway means rate limiting, auth enforcement, and TLS termination are reimplemented per-service via the shared nfr/ module rather than centralised at the edge.
Rejected Monolith × 4–5 coarse services × Serverless functions ×
D12
AUTH = stateless_jwt
Chosen ✓

Short-lived JWT access tokens (15 min) with 7-day refresh tokens via python-jose. Each service validates tokens independently using the shared secret — no round-trip to user_management per request, no session store. Separate JWT namespaces (distinct iss claims and secrets) for B2C users and B2B sellers prevent cross-namespace token reuse.

⚖ Trade-off accepted Immediate revocation is impossible without a denylist. A stolen access token stays valid for up to 15 minutes. The short TTL is a deliberate risk calibration — short enough to limit exposure, long enough to avoid excess refresh calls. Production systems add a Redis denylist for critical revocation (account takeover, admin force-logout).
Rejected Server-side sessions + Redis × OAuth2 / OIDC (over-engineered for capstone) × API-key-only auth ×
D13
RATE_LIMITING = slowapi_in_process
Chosen ✓

Per-IP rate limiting on auth endpoints using slowapi: 3/min on register, 5/min on login, 20/min on token refresh. This directly addresses the account-takeover NFR with no additional infrastructure. The guardrails service runs as a dedicated microservice (port 8009) rather than inline middleware, making AI I/O safety reusable across shopping_assistant, tool_calling_agent, and multi_agent_system without code duplication.

⚖ Trade-off accepted In-process counters do not aggregate across replicas — two instances of user_management each allow 5 logins/min from the same IP, doubling the effective limit. Production fix: storage_uri="redis://..." in slowapi config, or centralise enforcement at an API gateway (Kong, AWS API Gateway).
Rejected Kong API Gateway × nginx rate-limit module × AWS API Gateway × No rate limiting ×
