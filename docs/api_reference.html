System Architecture
Microservice Topology
The platform is built as a distributed microservice architecture where each service owns a bounded domain, exposes a FastAPI REST interface, and runs independently on a dedicated port. The React frontend communicates with services via HTTP through a single Axios client layer. Services share a PostgreSQL database with pgvector extension for semantic search (1,536-dimensional embeddings).

AI Architecture
RAG Pipeline — How AI Chat Works End to End
The AI shopping assistant uses a Retrieval-Augmented Generation (RAG) pipeline. This means it doesn't just run a language model — it first searches the product database semantically, injects relevant results, then generates a personalised response. This is why the answers reference real products with correct prices.
1
User sends natural-language query
Example: "wireless earphones good for running under ₹3000". Frontend calls :8002 POST /chat.
2
Guardrails check
:8010 screens the query for prompt injection, hate speech, and off-topic content before anything reaches the LLM. Blocked queries return a safe refusal immediately.
3
Embed the query with Azure OpenAI
The query is sent to text-embedding-3-small (Azure OpenAI). It returns a 1,536-dimensional floating-point vector capturing the semantic meaning of the query.
4
pgvector cosine similarity search
:8001 runs a vector similarity query against the products.embedding column. The top-K most semantically similar products are retrieved — not just keyword matches. A product titled "noise-cancelling sport buds" will surface for the running earphones query.
5
Personalisation from purchase history
:8004 fetches the user's past orders. Brands and categories the user has bought before are up-ranked in the result set so recommendations feel personal.
6
Prompt construction & GPT generation
A system prompt is assembled with the top product cards injected as context. GPT-4o-mini (Azure OpenAI) generates a conversational response that references the specific products, their prices, and why they fit the query.
7
Response returned with product pills
The API response includes both the natural-language reply and a structured products[] array. The frontend renders clickable product cards below the chat message.

User Management
localhost:8000 Authentication JWT Profiles
Why it exists: Every user-facing feature — saving addresses, tracking orders, personalised recommendations — requires knowing who is making the request. This service is the single source of truth for identity. It issues JWTs (JSON Web Tokens) that every other service verifies to protect data access.

What it achieves: Secure account creation, password-hash-based login, token issuance, profile management, and address book operations. A user logging in once receives a token they present to every subsequent API call, eliminating the need for re-authentication per service.

Product Catalogue

Why it exists: Products are the core data entity of an ecommerce platform. This service centralises all product data access — listings, search, filtering, individual product detail — so every other service (checkout, assistant, recommendations) queries one authoritative source rather than duplicating product logic.

What it achieves: Full-text and semantic search via pgvector cosine similarity on 1,536-dim embeddings (generated by text-embedding-3-small). Keyword search alone misses intent ("headphones for gym"); vector search understands meaning. Also provides category/brand browsing, price filtering, sorting, and paginated results.

Service 03 · Port 8002
Shopping Assistant
localhost:8002 RAG GPT-4o-mini Conversational AI pgvector
Why it exists: Keyword search forces users to know exactly what they want to type. A conversational AI assistant understands intent — users can ask vague, natural questions ("gift for a 10-year-old who likes space") and get relevant results. This is the highest-differentiation feature of the platform, directly increasing product discovery and conversion.

What it achieves: Maintains a persistent chat session, embeds each user message, retrieves semantically matching products from pgvector, personalises results using purchase history, and generates natural-language GPT responses with product cards. Falls back to Flan-T5-base (local model) if Azure OpenAI is unavailable.

Service 11 · Port 8010
Guardrails Service
localhost:8010 Content Moderation Prompt Injection Defense Safety
Why it exists: Any system that feeds user text into an LLM is vulnerable to prompt injection attacks (users crafting inputs that override system instructions), jailbreaks, and generating harmful content. The guardrails service intercepts all AI inputs and outputs to enforce a safety perimeter.

What it achieves: Pre-flight screening of user messages for injection attempts, hate speech, illegal content, and off-topic queries (not shopping-related). Post-generation screening of LLM output for policy violations. Logs all violations for audit. The shopping assistant and tool-calling agent both call this service before and after every LLM interaction.
MethodEndpointWhat it doesWhy it's needed

POST /guardrails/check-output Screen LLM-generated response before it's sent to the client Catches policy violations that slipped through the input check (e.g. model generating misleading price information)
POST /guardrails/check-relevance Determine if a query is shopping-related (vs. general knowledge, politics, etc.) Keeps the assistant focused on its purpose; off-topic queries get a polite redirect rather than consuming tokens on irrelevant generation
GET /guardrails/violations Paginated log of all detected violations (admin only) Audit trail for security review; helps identify patterns of misuse and tune guardrail sensitivity
PUT /guardrails/config Update guardrail sensitivity thresholds without redeployment Allows ops team to adjust false-positive/negative trade-off in response to observed violation patterns

External API Integrations
3 third-party services — Azure OpenAI for AI capabilities, Razorpay for payments, Shiprocket for logistics
External API 01
Azure OpenAI
GPT-4o-mini text-embedding-3-small REST API
Why it's required: The entire AI shopping assistant, semantic search, and tool-calling agent depend on two capabilities: (1) generating natural-language responses and (2) embedding text into vectors for similarity search. Azure OpenAI provides both as managed, enterprise-grade APIs with SLA guarantees, content filtering, and Indian data-residency options.

What it achieves: Powers all intelligent features of the platform. Without it, the search degrades to keyword-only (no semantic understanding), the chat assistant returns template responses, and recommendation personalisation loses its language-model quality. A local Flan-T5-base fallback exists but produces significantly lower-quality results.
Models Used
GPT
GPT-4o-mini (GPT-5.4-mini deploy)
Chat Completion · gpt-4o-mini
Generates conversational shopping assistant responses, synthesises multi-agent outputs, and powers tool-calling decision-making. Chosen for its balance of quality and speed at low cost — complex reasoning isn't required for most shopping queries.
Why this model
Fast inference (~800ms), supports function-calling JSON mode, low per-token cost for high-frequency chat workloads. GPT-4-level quality not needed for product recommendation responses.
EMB
text-embedding-3-small
Embeddings · 1,536 dimensions
Converts product descriptions and user queries into 1,536-dimensional float vectors for pgvector cosine similarity search. Every product in the catalogue has a pre-computed embedding; queries are embedded at search time for comparison.
Why this model
State-of-the-art retrieval quality in a compact 1,536-dim space. Cheaper than text-embedding-3-large (3,072 dims) with minimal quality loss for product search. Embeddings are computed once per product and stored, so ongoing cost is only for query embedding.
API Operations
POST /chat/completions
Main LLM call — sends conversation history + product context as system message; receives assistant reply. Called by shopping_assistant and tool_calling_agent services for every user message.
POST /embeddings
Generates the 1,536-dim vector for a product or query text. Called during: product creation/update (product embedding), user search queries (query embedding), bulk product seeding.
Function Calling (tool_use)
JSON schema function definitions passed to the chat completion API allow GPT to decide which internal tools (search_products, add_to_cart, etc.) to invoke and with what arguments.
External API 02
Razorpay
UPI Cards Net Banking HMAC-SHA256 Webhooks
Why it's required: Building a payment processor from scratch requires PCI-DSS compliance, banking partnerships, and fraud detection — all extremely expensive and time-consuming. Razorpay is an Indian payment gateway that provides all of these out of the box, supports UPI, cards, net banking, and wallets, and is trusted by thousands of Indian businesses.

What it achieves: Secure payment collection supporting all major Indian payment methods. The server-side order creation + client-side widget + server-side signature verification flow ensures the payment amount cannot be tampered with by the client. HMAC-SHA256 signature verification guarantees that webhook events genuinely came from Razorpay.
Payment Flow
1
Server creates Razorpay order
:8009 calls POST /v1/orders with amount in paise, currency INR, and receipt (our order ID). Razorpay returns a razorpay_order_id.
2
Client-side payment widget
Frontend initialises Razorpay.js with the razorpay_order_id. User completes payment (UPI, card, etc.) in the Razorpay-hosted widget. On success, Razorpay returns payment_id, order_id, signature.
3
Server verifies signature
:8009 computes HMAC-SHA256(razorpay_order_id + "|" + razorpay_payment_id, secret_key) and compares against the received signature. Mismatch → reject. Match → proceed to order confirmation.
4
Async webhook confirmation
Razorpay sends a payment.captured webhook (also HMAC-SHA256 verified) as a reliable secondary confirmation. Used to handle edge cases where the client never called /verify (browser crash, network loss).
API Operations Used
POST /v1/orders
Create a Razorpay order. Server-side only — ensures amount cannot be manipulated by client-side JS.
GET /v1/payments/{id}
Fetch full payment details for admin order reconciliation and receipt generation.
POST /v1/payments/{id}/refund
Issue a full or partial refund to the buyer's original payment method. Called on order cancellation and return approval.
Webhook: payment.captured
Async event fired when payment is captured. Secondary confirmation path — updates order to "payment_confirmed" if /verify was not called.
Webhook: payment.failed

External API 03
Shiprocket
Shipping Aggregator Multi-courier JWT Auth Webhooks
Why it's required: Shipping logistics requires integrating with multiple courier companies (BlueDart, Delhivery, DTDC, Ekart, etc.) — each with their own API, label format, and tracking system. Shiprocket is an aggregator that provides a single API across 17+ courier partners, automatic courier selection based on weight/pincode/cost, and a unified tracking system.

What it achieves: Single-API shipment creation for all courier partners, automatic AWB (Air Waybill) number generation, pincode serviceability checks, live tracking aggregated from multiple couriers, label generation, and delivery webhook events. The JWT token (23h TTL) is cached in the payment_shipping service to avoid re-authenticating on every call.
API Operations Used
POST /v1/external/auth/login
Authenticates with Shiprocket using email+password; returns a JWT valid for 23 hours. Cached by payment_shipping service to avoid re-authentication on every shipment creation.
POST /v1/external/orders/create/adhoc
Create a shipment order in Shiprocket. Sends buyer address, product weight/dimensions, and seller pickup address. Shiprocket auto-selects the cheapest/fastest courier and returns an AWB tracking number.
GET /v1/external/courier/serviceability
Check if a delivery pincode is serviceable and get estimated delivery dates per courier. Shown to user at checkout so they can see when their order will arrive before paying.
GET /v1/external/courier/track/awb/{awb}
Live tracking for an AWB number — returns current status, location, and estimated delivery date. Used by order tracking page and AI tracking tool.
POST /v1/external/orders/cancel
Cancel a shipment before courier pickup. Required when an order is cancelled after a Shiprocket order was already created but before physical pickup.
GET /v1/external/courier/generate/label
Generate a printable shipping label PDF for the seller to attach to the package. Includes barcode, address, and AWB number.
Webhook: shipment_status
Push notification from Shiprocket when package status changes (pickup → in-transit → out-for-delivery → delivered). Updates order status in real-time without polling.
